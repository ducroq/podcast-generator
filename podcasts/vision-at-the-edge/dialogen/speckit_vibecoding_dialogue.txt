Marc: [neutral] The basic idea is: before you write any code, you write a specification. A document that describes what you want to build and why.

Marc: [thoughtful] The twist is that the specification isn't just documentation—it becomes the source of truth. You don't throw it away after you start coding. The spec directly drives the implementation.

Sven: [skeptical] I don't get it. How is writing a document before coding any different from... writing a document before coding?

Marc: [emphatic] Because with AI, you can actually enforce it. The AI reads your spec, generates a plan, breaks it into tasks, and then implements exactly what you specified. If you didn't spec it, it doesn't get built.

Marc: [warm] Precisely. And there's this toolkit from GitHub called Spec-Kit that formalizes the whole workflow.

Marc: [thoughtful] Okay, so there are five main steps. First, you create what they call a constitution.

Marc: [neutral] Kind of! It's a document that defines your project's principles. Things like "we prioritize privacy" or "we keep things simple" or "we always write tests first."

Lisa: [curious] And the AI reads this?

Marc: [emphatic] Every time. So when it makes decisions during implementation, it checks against those principles. It's like giving the AI a value system.

Sven: [thoughtful] Huh. That's actually clever. So it won't add a bunch of tracking code if your constitution says privacy-first.

Marc: [warm] Exactly. Then step two: you specify the feature. But here's the key—you focus on the what and why, not the how.

Lisa: [curious] Can you give an example?

Marc: [neutral] Sure. Instead of saying "create a React component with useState hooks that fetches from this API," you'd say "users should be able to see their recent orders so they can track deliveries."

Sven: [skeptical] But that's so vague! How does the AI know what to build?

Marc: [thoughtful] That's actually the point. The vagueness forces you to think about what you actually need, not how to implement it. The implementation details come later, in the planning phase.

Lisa: [warm] So you're separating concerns.

Marc: [emphatic] Exactly. First you nail down requirements. Then you pick technology. Then you break it into tasks. Then you implement.

Marc: [neutral] All true. But here's the counter-argument: with AI, you can regenerate implementation in minutes, not months. If your spec is wrong, you update it and regenerate.

Lisa: [surprised] So the spec is a living document, not carved in stone.
